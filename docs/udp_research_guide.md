# Comprehensive Research Guide: Zynq-7000 Bare-Metal UDP Troubleshooting

Since the standard bare-metal network initialization (lwIP + GIC interrupts + Timers) is still resulting in a "Request timed out" during ping, the issue lies in a deeper edge-case. 

Here is a comprehensive list of specific areas, search terms, and hardware/software layers you should research to definitively root-cause this issue.

---

## 1. Vitis 2024.1 SDT (System Device Tree) Interrupt Routing
The new Vitis SDT architecture dynamically generates hardware definitions differently than the older "classic" Vitis. 
* **The Theory:** The MAC interrupt ID defined in the generated `xparameters.h` might not match the actual hardware interrupt line connected to the GIC (Generic Interrupt Controller) in your Vivado block design. If the MAC raises IRQ 54, but the software is listening on IRQ 77, packets will be silently dropped.
* **What to research/check:**
  * Search term: `"Vitis SDT lwIP interrupt not firing"`
  * Search term: `"Zynq-7000 GEM interrupt ID mismatch System Device Tree"`
  * **Action:** Open your Vivado Block Design, trace the `enet0` interrupt line to the `IRQ_F2P` / GIC block, and verify the IRQ number matches the `XPAR_XEMACPS_0_INTR` definition in your `xparameters.h`.

## 2. PC Network Routing & Virtual Adapters (Windows Specific)
Windows has complex routing rules when multiple network adapters exist (VPNs, WSL, VirtualBox, Wi-Fi).
* **The Theory:** The PC is sending the ping/ARP requests out of the wrong physical or virtual interface, or the replies are being routed to a dead virtual adapter.
* **What to research/check:**
  * Search term: `"Windows ping Request timed out but ARP fails multiple adapters"`
  * **Action:** Run `ipconfig /all` and ensure `192.168.1.100` belongs ONLY to the physical Ethernet adapter connected to the Zedboard. Disable all other virtual adapters (like WSL or VirtualBox) temporarily.

## 3. Cache Coherency and DMA (Direct Memory Access)
The Zynq-7000 ARM Cortex-A9 CPU uses L1 and L2 caches. The Ethernet MAC uses DMA to write packets directly into DDR RAM.
* **The Theory:** The MAC successfully writes the incoming ARP packet to RAM via DMA, but the ARM CPU reads from its L1/L2 cache instead of RAM, seeing stale data (empty packets). 
* **What to research/check:**
  * Search term: `"Zynq-7000 bare-metal lwIP cache coherency issues"`
  * Search term: `"Xil_DCacheInvalidateRange lwIP xemacpsif_dma"`
  * **Action:** Research how memory mapped to the Ethernet DMA descriptors is configured in the MMU. Often, a specific memory region must be marked as "uncached" using `Xil_SetTlbAttributes()` to prevent these exact DMA/CPU race conditions.

## 4. Marvell PHY Initialization and Auto-Negotiation
The Zedboard uses a specific physical layer chip (PHY), usually a Marvell 88E1518.
* **The Theory:** `lwIP`'s default `xemac_add` routine tries to auto-negotiate with the PHY via MDIO. If the negotiation fails or takes too long, the MAC is never fully linked, or the link speed (10/100/1000) is misconfigured between the MAC and the PHY.
* **What to research/check:**
  * Search term: `"ZedBoard Marvell PHY baremetal lwIP initialization"`
  * Search term: `"lwIP eth_mac_init PHY auto-negotiation hang"`
  * **Action:** Research how to manually force the PHY speed (e.g., 100 Mbps Full Duplex) via MDIO registers instead of relying on auto-negotiation, which is notoriously flaky on custom bare-metal stacks.

## 5. Memory Pool Exhaustion in `lwipopts.h`
`lwIP` is highly configurable via a file called `lwipopts.h`.
* **The Theory:** When a packet arrives in the interrupt handler (`emacps_recv_handler`), it attempts to allocate memory (`pbuf_alloc`). If the memory pool sizes are too small, the allocation fails, and the packet is immediately dropped before the software ever sees it.
* **What to research/check:**
  * Search term: `"lwIP PBUF_POOL_SIZE MEM_SIZE tuning for Zynq"`
  * **Action:** Inspect the `lwipopts.h` file generated by Vitis. Look for `MEM_SIZE`, `PBUF_POOL_SIZE`, and `MEMP_NUM_PBUF`. Increase them significantly (e.g., `MEM_SIZE` to `524288`) to rule out out-of-memory drops.

## 6. MAC Hardware Filtering (Broadcast Packets)
ARP relies on broadcast packets (`FF:FF:FF:FF:FF:FF`).
* **The Theory:** The Gigabit Ethernet Controller (GEM) hardware has strict MAC filtering rules. If the `nwcfg` (Network Configuration) register is set to drop broadcast packets, ARP will never reach the software.
* **What to research/check:**
  * Search term: `"Zynq UG585 GEM network configuration register broadcast drop"`
  * **Action:** Read Chapter 16 of the Zynq-7000 Technical Reference Manual (UG585). Specifically, look at the `XEMACPS_NWCFG_OFFSET` register bits to see if broadcast packet reception is disabled.
