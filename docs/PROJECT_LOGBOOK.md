## 15/07/2026
### Opal Kelly OV7670 FPGA Camera Stabilization & Clocking
- **Camera Blackout & Clock Domain Synchronization**: Diagnosed an issue where `camera_top.v` would lock up completely after reading a single frame. Discovered that the asynchronous software `reset` trigger (from Python) was causing metastability in the 100MHz SDRAM arbiter state machine. Implemented dual-rank flip-flop synchronizers (CDC) on the SDRAM arbiter inputs, and added a safe 10-clock cycle synchronous reset recovery delay before allowing write-enables.
- **PLL Configuration Bug (CY22393)**: The camera crashed consistently after exactly ~1 millisecond (23,172 clocks). Mined the Opal Kelly XEM3010 User's Manual and discovered that the Python script was configuring the 24MHz camera clock on PLL **Output 0**, but the camera's `clk2` (P9) was physically hardwired to PLL **Output 2**. Because Output 2 was unconfigured, the CY22393 defaulted to bypassing the PLL, blasting the raw 48MHz reference crystal into the OV7670. The camera survived for 1ms on a 200% overclock before its internal silicon crashed. Fixed `grab_frame.py` to route the 24MHz clock to Output 2.
- **I2C SCCB Lockup & Hardware Reset Sequencing**: Even with a stable 24MHz clock, the camera failed to output frames (`VSYNC=0`). Realized that `v_sup` (mapped to the camera's physical RESET pin) was held high (out of reset) instantly upon FPGA power-up. Because the camera booted up for several seconds *before* the Python script configured the 24MHz clock, the camera's internal state machine permanently glitched. When the I2C configuration was finally sent, the locked-up camera ignored all commands. Fixed by mapping `v_sup = ~reset[0]` in Verilog, and updating `grab_frame.py` to send a clean 10ms hardware reset pulse *after* the PLL clock stabilizes, ensuring the SCCB interface cleanly accepts the I2C configuration.

## 15/07/2026
### FPGA Camera Diagnostics & Architecture
- **VirtualBox USB Grab Bug**: Diagnosed why the Opal Kelly XEM3010 device was failing to open on the Windows host. The VirtualBox USB Passthrough driver was actively grabbing the FPGA to forward it to the Linux VM, hiding the hardware serial number from the host OS.
- **I2C Configuration Endpoint Bug**: Discovered a major bug in `camera_top.v` where the Python script's configuration check (`WO_CONF = 0x22`) was mistakenly hardwired to the FIFO's `empty` flag instead of the actual `done` signal. This caused the script to skip camera I2C configuration entirely. Fixed by mapping `done` to `0x20`.
- **USB 2.0 PipeOut FIFO Overflow**: Found that streaming 640x480 RGB565 at 24MHz (30FPS) directly into a tiny 4KB Block RAM FIFO caused immediate overflows and dropped pixels, leading to severe image tearing and color corruption. The USB 2.0 host polling overhead could not drain the FIFO fast enough.
- **Clock Division vs SDRAM Controller**: Evaluated building a complex SDRAM controller to buffer the frame versus simply slowing the camera clock. Prevented the USB overflow by modifying the OV7670's `CLKRC` register to divide the internal pixel clock by 4 (`0x03`), dropping the framerate to ~7.5 FPS. This successfully stabilized the image stream over USB without requiring an SDRAM controller.
- **PLL Clock Physical Mapping Bug**: Diagnosed an issue where the camera produced no pixel clock (PCLK dead) despite successful I2C configuration. Discovered that the XEM3010 physically hardwires its onboard SDRAM clock exclusively to FPGA pin `N9` (driven by CY22393 PLL Output 1), while we had incorrectly assigned the 24MHz camera clock to that output. Swapped the Python PLL configuration so Output 1 (`N9`) outputs 100MHz for the SDRAM, and Output 0 (`P9`) outputs 24MHz for the camera `xclk`. Updated the UCF constraints to explicitly declare both clocks with their correct physical routing and timing specifications (`TS_clk1=100MHz`, `TS_clk2=24MHz`).
- **Clock Generator VCO Range Bug**: Discovered that the CY22393 PLL was completely failing to lock on the 24MHz `xclk` because the Python configuration requested a VCO frequency of 48MHz (which violates the strict 100-400MHz operational bounds of the chip). Recalculated the PLL multipliers in `grab_frame.py` to use a 240MHz VCO divided by 10, perfectly resolving the instability.
- **Asynchronous FIFO Reset CDC Bug**: Diagnosed a major cross-clock domain hazard where the host's asynchronous `reset` signal was driving the synchronous `rst_n` pin of the 100MHz SDRAM state machine. Implemented a dual-rank flip-flop CDC synchronizer in `sdram_arbiter.v` to safely buffer the reset signal into the 100MHz domain, eliminating metastability lockups.
- **OV7670 PWDN Shutdown Cascade**: Uncovered a massive, cascading pipeline failure that caused the Python `ReadFromBlockPipeOut` loop to hang forever. Discovered that the FPGA `v_sup` pin (mapped to the camera's active-high Power Down `PWDN` pin) was hardcoded to `1'b1`. This actively commanded the camera to enter deep sleep immediately upon startup. The camera would output exactly 23,172 clocks (~1 millisecond) before completely dying, meaning it never emitted a single `VSYNC` or `HREF`, which kept the SDRAM arbiter starved and locked up the USB FIFO pipeline. Fixed by setting `v_sup = 1'b0` to wake the camera.

### ML Vision Inference & Architecture
- **Inference Pipeline Framework**: Developed the foundational scripts for real-time inference on the host PC (`host_inference_loop.py`). The pipeline ingests 30FPS compressed UDP frames from the iPhone, instantly decodes them, passes them through a PyTorch model for `(x, y)` inference, calculates positional error based on targets, and pushes packed 16-bit binary targets directly to the STM32 over serial.
- **Color Space Bug Fix**: Identified and resolved a critical bug where OpenCV's `BGR` default decoding would invert the red and blue color channels during real-time inference (since the PyTorch model was trained on `RGB` via PIL). Inserted an instantaneous `cv2.cvtColor` step into the pipeline.
- **Temporal Dataset Restrictions**: Refactored the ML training scripts (`train_expert_tracker.py` and `train_expert_tracker_subset.py`) to enforce strictly temporal train/test splits. Instead of randomized shuffling, the model trains exclusively on earlier chronological sequences (e.g., first 80%) and validates on completely unseen future contiguous sequences to prevent temporal data leakage.
- **Unsupervised Marker Identification Test**: Explored identifying physical control markers via pure Unsupervised Machine Learning (K-Means spatial-color clustering). Developed a multi-masking HSV strategy to isolate Red, Green, Grey, and Black markers. Discovered that preserving the low-saturation Grey/Black markers forced the mask to absorb over 92,000 background shadow pixels per frame, slowing K-Means to a crippling 5 seconds per frame (0.2 FPS). Concluded that purely unsupervised thresholding lacks the spatial context to reliably discard shadows in real-time. Documented this finding in `archived_ideas.md` and pivoted towards Transfer Learning.
- **FPGA Edge Inference Roadmap**: Logged a project goal in the Vision TODO to ultimately synthesize the final ML model weights into logic gates using Vitis HLS/hls4ml for native FPGA execution, eliminating host PC communication latency entirely.

### Control Systems Overhaul & Data Collection Robustness
- **Virtual Wall Oscillation Fix**: Discovered that the emergency "Virtual Wall" was causing the ball to aggressively ping-pong off the edges. The wall was teleporting the setpoint to `(0,0)`, which created a massive 71mm error and catapulted the ball at the maximum 12.5-degree tilt. Softened the wall to only push the target 15mm inwards (`56mm`), providing a firm but controlled restorative tilt.
- **Slew Rate Limiter**: Implemented a Setpoint Slew Rate Limiter in `PIDControllers.cpp`. The state machine now instantly outputs discrete targets, but the PID controller internally ramps the setpoint at a maximum of `80mm/s`. This prevents sudden massive target jumps from generating violent error spikes.
- **Derivative Kick Elimination**: Diagnosed severe control loop fighting caused by the new Slew Rate Limiter. Because the target was actively moving, taking the derivative of the Error (`Target - Ball`) caused the D-term to think the ball was flying at 80mm/s. Rewrote the D-term to calculate Derivative on Measurement (raw ball velocity) instead of Error, perfectly smoothing the movement.
- **EWMA Settling Filter**: Removed the unreliable "3-second timer" for detecting when the ball reached a target. Replaced it with an Exponentially Weighted Moving Average (EWMA) filter on the X and Y errors. Adjusted the alpha weight to `0.015` (requiring ~8 seconds of continuous deadband stability) to mathematically guarantee the ball is at rest before generating the next target.
- **Phase Start Synchronization**: Discovered that Phase 2 (Patterns) and Phase 3 (Sweeps) were blindly tracing their trajectories without waiting for the ball to arrive at the starting points. Added a `waiting_at_start` boolean to force the state machine to wait for EWMA stability at the start of every pattern and sweep, drastically improving the cleanliness of the ML dataset.
- **Touchscreen ADC Debouncing**: Resistive touchscreens occasionally drop readings for 1-2 milliseconds due to ADC noise or the ball slightly bouncing. Added a `last_ball_detected_ms` tracker to gracefully debounce signal drops. The state machine now gives the ball a 1.5-second grace period before triggering an emergency Phase 0 recovery.
- **End-of-Run Finish Sequence**: Modified the main `BallBalancingBot.ino` loop to capture the `is_done` flag. When all data collection phases are complete, the robot now automatically tilts to a steep 12-degree pitch to roll the ball completely off the platform, and permanently locks the stepper motors.
- **Telemetry Upgrades**: Updated the Python `iphone_data_logger.py` and real-time `plot_log.py` scripts to dynamically parse an expanded binary struct. 
### Motor Calibration & Touchscreen Inverse Kinematics Fixes
- **Motor Initialization**: Updated `MotorControl.cpp` to use the correct user-calibrated raw steps (`STEPS_TO_ORIGIN` 475, 425, 505) instead of approximate angles to guarantee a perfectly flat horizontal plane upon startup. Fixed the `ENA` pin mapping on the STM32 to `PD0` to correctly power on the drivers.
- **Removed Blocking PID Delay**: Removed an erroneous `delay(10)` from the `pid_balance()` timeout condition. This delay was starving the non-blocking `AccelStepper` `run()` function of CPU cycles, forcing the motors to crawl back to horizontal at a sluggish 100 steps/sec (which appeared as "jittering/locking"). The motors now return to perfectly horizontal instantly.
- **PID Speed Limits & Jitter Fix**: Drastically increased `AccelStepper` limits (Max Speed 6000, Accel 25000) for instant, rapid flicking in response to PID outputs. Uncommented the `deriv[i]` constraint in `PIDControllers.cpp` to heavily filter out touchscreen ADC noise and eliminate derivative motor buzz when the ball is stationary.
- **Touchscreen 180-Degree Coordinate Alignment**: Diagnosed a severe Inverse Kinematics runaway positive feedback loop where pressing the Top Left corner of the screen caused the plate to tilt *towards* the stylus. Discovered the physical touchscreen is mounted 180 degrees rotated relative to the mathematical model (Motor C is physically Top Left, but mathematically Bottom Right). Fixed by digitally inverting the `x_mm` and `y_mm` mappings in `Screen.cpp` by `*-1`.
- **Motor Homing Directionality**: Discovered that physically flipping the A and B motor wires had inverted the mathematical definition of a "Positive" step direction to mean DOWN. Fixed the startup grinding issue by commanding the motors to move in the negative direction (`-STEPS_TO_ORIGIN`) during `home_motors()`, successfully driving them UP away from the physical hardstops.
- **PID Derivative Uncapping**: Identified a critical bug preventing the ball from stopping: the `deriv` term was hard-capped at `55`. This neutered the D-term's ability to brake fast-moving balls (since a typical roll speed is 150+ mm/s). Expanded the constraint to `[-800, 800]` to restore full shock-absorbing power.
- **Live Telemetry Visualisation**: Completely overhauled `plot_log.py` using `pyserial` threading and `matplotlib` `FuncAnimation` to stream and plot real-time PID errors directly from the COM port, greatly improving the tuning feedback loop.
- **Diagnosed Asymmetric Steady-State Error**: Utilized the new live plotter to definitively prove the ball was trapped exclusively in the `+X, +Y` quadrant, confirming a severe physical unlevel calibration. Concluded that an active Integral (`ki = 0.25`) term is absolutely mandatory to automatically wind up and overcome this gravity slope.
## 16/07/2026
### YOLO-Pose Augmentation & Geometry Reconciliation
- **YOLO Flipping Bug**: Discovered a critical issue where default YOLOv8 augmentations (`fliplr=0.5`) were silently corrupting our dataset. Because our keypoints had strict spatial semantics (e.g., Keypoint 0 = Top-Left), horizontal flips essentially mirrored the image without mirroring the semantic labels, severely penalizing the model during training. Explicitly disabled `fliplr=0.0` and `flipud=0.0` in `train_yolo.py`.
- **CAD Plate vs. Resistive Touchpad Dimensions**: Solved an elusive homography projection bug that was "shrinking" our targets and ball mapping inward. Reconciled the fact that the 4 YOLO-Pose keypoints visually track the outer boundaries of the purple acrylic plate (`200.0 x 146.93 mm`), while the telemetry data (`touch_x`, `touch_y`) and target coordinates strictly exist within the inner resistive touchpad (`187.5 x 142.0 mm`).
- **Homography Coordinate Shifting**: Adjusted `generate_unified_pose_dataset.py` to use the true `200.0 x 146.93 mm` boundaries for the homography matrix calculation. Shifted the internal Target coordinates into this Plate Space by adding half the difference (`PAD_OFFSET_X`, `PAD_OFFSET_Y`) and properly mapping the ball by adding half the Plate dimensions. This flawlessly aligned the telemetry with the visual bounding box.
- **360-Degree Rotation Invariance**: Realized that restricting YOLO to a single viewing angle was limiting. Rewrote the dataset generation pipeline to apply random arbitrary rotations (0 to 360 degrees) to the raw 02_silver dataset images.
- **Physical Keypoint Anchoring**: To ensure the semantic meaning of the keypoints survived 360-degree rotations, we moved away from image-relative labels (like "Top-Left"). We physically anchored Keypoint 0 to the corner anti-clockwise from the motor lip. By applying OpenCV's affine rotation matrix to both the image pixels and the 4 physical keypoint coordinates simultaneously, the model is now trained to track the physical geometry of the board regardless of the camera's orientation.
- **Intrinsic Coordinate Invariance (ResNet)**: We originally thought applying PyTorch `RandomAffine` or `RandomPerspective` to the ResNet Expert Tracker without modifying the target labels was a bug. However, we realized that the labels (`touch_x`, `touch_y`) are *intrinsic* to the physical board! If the camera translates, zooms, or rotates (extrinsic changes), the physical position of the ball on the board NEVER changes. Therefore, applying these geometric augmentations *while keeping the labels identical* is mathematically perfect! It brilliantly forces the CNN to learn camera-invariance by locating the purple shape of the platform's outline to calculate relative position, rather than memorizing absolute pixel locations.
- **The Only Forbidden Augmentation**: `RandomHorizontalFlip` remains strictly forbidden for both models. Flipping the image mirrors the board (e.g. swapping the left/right targets), creating a physically impossible hardware layout that destroys the intrinsic coordinate system.
